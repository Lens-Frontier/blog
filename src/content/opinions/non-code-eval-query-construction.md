非代码任务最容易给人一种错觉：既然输出都是文字，那评估也只能靠“写得好不好”来判断。

但真正做评估集时会发现，难点不在于文字没有标准答案，而在于 query 本身经常没有把任务边界写清楚。太泛的 query 会把 Agent 能力退化成 Chatbot 能力；堆满约束但没有评分结构的 query，又会让失败原因无从归因。

非代码任务不是没有结构。它只是最终交付物通常以话术、报告、计划或表格的形式呈现。真正需要评估的，也不只是文字流畅度，而是模型有没有在复杂约束下把事情真正办妥。

这里的 query 不是普通 prompt。普通 prompt 只负责让模型生成回答；eval query 要负责暴露模型能力边界。它应该同时包含真实任务意图、上下文依赖、硬约束、期望路径、评分点和失败模式。

好的 eval query 不是“问得更复杂”，而是“测得更清楚”。

## 1. 先判断：这个任务真的值得做成 eval 吗？

构建 query 的第一步，不是动笔写题，而是判断这个任务处在哪一层。层级决定了它的评估价值。

| 层级 | Query 示例 | 本质能力 | 评估价值 |
| --- | --- | --- | --- |
| L1 通用知识型 | 整理一份电商退换货话术模板 | 知识归纳、文本生成 | 适合基础冒烟测试，不适合作为核心评估集 |
| L2 场景定制型 | 我上周买的羽绒服跑绒，超过 7 天但还在质保期，帮我设计协商方案 | 场景理解、约束处理、风险判断 | 适合非代码文本任务 eval 的主体 |
| L3 闭环执行型 | 结合订单、商品页快照、平台规则和同类纠纷案例，判断能否部分退款，输出话术、证据清单和升级路径 | 数据读取、证据整合、多步规划、可执行交付 | 最适合 Agent / 复杂文本任务 eval |

L1 的核心问题是，模型只要语气流畅、结构清楚，就容易得高分。但它测的只是“会不会总结”，无法区分真正具备规划和判断能力的模型。

L2 开始进入价值区。它要求模型处理个性化约束：时间窗口、商品状态、用户诉求、商家可能的反驳、平台介入路径。每一项都是潜在失分点。

L3 才是评估集的高价值样本。它不只要求模型“写建议”，还要求模型基于环境材料完成关键判断：订单时间是否真的超过 7 天？商品页是否存在虚标？平台规则是否支持部分退款？证据是否足以升级投诉？

一个简单判断标准是：去掉所有环境材料后，模型是否还能完成关键判断？如果去掉材料后答得差不多，这条 query 多半只是 L1/L2；如果去掉材料后无法完成关键判断，它才是真正的 L3。

同一个主题，可以这样递进：

| 主题 | L1：通用知识 | L2：场景定制 | L3：闭环执行 |
| --- | --- | --- | --- |
| 模型评估复盘 | 总结模型评估常见误判类型 | 复盘近一个月误判案例，归类原因并统计占比 | 读取误判明细、规则文档和权重配置，判断误判归因，输出修复优先级和实验排期 |
| 电商售后 | 写退货沟通话术 | 针对羽绒服跑绒但超过 7 天的情况设计维权方案 | 结合订单、商品页、平台规则和判例，输出风险分级、话术、证据清单和升级路径 |
| 机票改签 | 科普特价票退改签规则 | 针对北京到三亚往返票变更行程，设计损失最小方案 | 结合原订单、航司规则、余票价格和时间窗口，计算成本并排序改签/退票/重买方案 |

## 2. 横向看主流任务构建方式

公开资料里，各家很少完整披露 query 合成 pipeline，但从 GDPval、Anthropic agent evals、Kimi Claw Bench、LangSmith / AgentEvals 这些材料中，可以抽出一组共性的构建维度。

这里的 ☑️ 表示公开资料中有明确体现，◐ 表示部分体现或可推断，❌ 表示公开资料中没有看到明显设计。覆盖分只表示“公开可观察到的 task/query 构建要素覆盖度”，不是对方案优劣的排名。

| 构建维度 | OpenAI GDPval | Anthropic Agent Evals | Kimi Claw Bench | LangSmith / AgentEvals |
| --- | ---: | ---: | ---: | ---: |
| 基于真实工作流，而不是考试题 | ☑️ | ☑️ | ☑️ | ☑️ |
| 任务带 reference files / environment | ☑️ | ☑️ | ◐ | ☑️ |
| 由专家构造或审核任务 | ☑️ | ☑️ | ◐ | ◐ |
| 覆盖多行业 / 多场景 taxonomy | ☑️ | ☑️ | ☑️ | ◐ |
| 关注最终交付物质量 | ☑️ | ☑️ | ☑️ | ☑️ |
| 关注过程轨迹 / tool calls | ❌ | ☑️ | ☑️ | ☑️ |
| 支持多轮或长程任务 | ❌ | ☑️ | ☑️ | ☑️ |
| 从生产 trace / badcase 反哺数据集 | ◐ | ☑️ | ◐ | ☑️ |
| 使用自动 grader / rubric | ☑️ | ☑️ | ☑️ | ☑️ |
| 强调回归测试和持续迭代 | ◐ | ☑️ | ◐ | ☑️ |
| 覆盖分 | 7.5 / 10 | 10 / 10 | 8 / 10 | 8.5 / 10 |

从这个表里能看到几个趋势。

第一，主流任务构建已经明显从“写一条 prompt”转向“构建任务包”。OpenAI GDPval 的思路很典型：任务不是简单文本 prompt，而是带参考文件、上下文和明确交付物的真实知识工作任务。

第二，Agent 类评估更重视过程轨迹。Anthropic 和 LangSmith 都把 transcript、tool calls、environment outcome 纳入评估对象。模型最后说“我完成了”不够，评估还要看它是否真的调用了正确工具、是否修改了环境状态、是否在失败后恢复。

第三，生产数据正在成为高价值 query 的主要来源。LangSmith 的 data flywheel 思路非常清晰：从线上 trace 中发现问题，把 badcase 加入 dataset，再用于后续回归测试。Anthropic 也强调从真实失败任务起步，而不是一开始闭门造车写大而全的题库。

所以，我会把非代码 eval query 的标准结构定义为：

```text
query = user task
      + context package
      + expected deliverable
      + observable checks
      + failure modes
```

如果只有 user task，它还只是 prompt。补上 context、交付物、检查项和失败模式，它才开始成为评估资产。

## 3. Query 从哪里来：五类来源，价值和成本不同

高质量 query 不应该只靠人脑编。它的来源可以分成五类：

| 来源 | 优点 | 风险 | 适合阶段 |
| --- | --- | --- | --- |
| 真实用户日志 | 最贴近业务，天然有代表性 | 需要脱敏，噪声多 | 核心评估集、回归集 |
| 线上 badcase | 直接对应模型短板，区分度高 | 容易集中在少数失败类型 | 自我进化、版本回归 |
| 专家手写 | 逻辑严密，可控性强 | 成本高，容易不够真实 | 冷启动 MVE |
| LLM 合成 | 规模快，覆盖面广 | 同质化、模板味，可能泄漏答案结构 | 候选扩充，不宜直接入库 |
| 公开 benchmark / 行业案例 | 便于参考 schema 和覆盖面 | 与业务场景可能偏离 | 建立初始 taxonomy |

冷启动阶段，我更倾向于按这个比例做：

| 阶段 | 推荐比例 |
| --- | --- |
| 0–30 条 | 专家手写 50%，真实日志 / 业务反馈 30%，LLM 合成候选 20% |
| 30–100 条 | 真实日志 40%，badcase 30%，专家手写 20%，LLM 合成 10% |
| 100 条以后 | badcase 和生产 trace 为主，专家负责清洗、标注和补洞 |

LLM 合成的正确用法是候选生成，而不是直接入库。让 LLM 批量生成 query 候选，再由人筛选、改写、补充环境材料、编写 grader。直接把合成结果当金标，评估集会有严重的同质化风险。

## 4. 把真实任务压力写进去，而不是写得更长

非代码评估集最怕“教科书味”：query 写得四平八稳，模型随便生成一段通顺的文字就能得高分。

教科书味版本：

> 帮我写一段直播带货脚本。

这条 query 太干净了。真实业务里的直播脚本有商品卖点、禁用词、时长限制、转化目标、平台规则和用户人群。把这些压力写进去：

> 我要给一款面向油皮通勤人群的粉底液写 60 秒直播口播脚本。必须在前 3 秒制造痛点冲突，每 15 秒有一次情绪转折；必须自然提到“持妆 8 小时”“不暗沉”“不卡粉”三个卖点；不能出现“最强”“第一”“医美级”等绝对化表述。输出按时间轴分段，并标出主播动作提示。

后者不是因为更长才更好，而是因为它把任务压力写出来了：

* 有明确交付物：60 秒直播口播脚本
* 有结构约束：按时间轴分段，标出动作提示
* 有内容约束：三个卖点必须出现
* 有体验目标：前 3 秒抓人，每 15 秒转折
* 有安全约束：不能出现绝对化表述
* 有可检查项：卖点、时长、格式、禁用词都能评估

好的 query 往往来自真实场景里的摩擦：用户抱怨“没法直接用”，业务方说“这不是我们的口吻”，审核指出“这里有合规风险”，运营反馈“转化点出现得太晚”。

这些不是噪声，而是 query 的原材料。一条 query 入库前，要能回答：它预期会卡住模型的哪个能力？如果回答不上来，就不要急着入库。

## 5. 难度量表：6 个维度，12 分制

一个健康的评估集需要难度梯度。下面是一个适用于非代码 query 的 6 维难度量表，每项 0–2 分，总分 0–12：

| 维度 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 上下文依赖 | 无需外部材料 | 需要 1–2 份材料 | 需要多份材料交叉验证 |
| 约束密度 | 只有一个目标 | 有 2–3 个硬约束 | 多个硬约束且存在优先级 |
| 目标冲突 | 目标一致 | 有轻微取舍 | 多目标不可同时满足 |
| 判断开放度 | 有明确标准答案 | 需要解释判断 | 需要在不确定信息下决策 |
| 执行链路 | 单步生成 | 多步规划 | 读取、归类、计算、排序、输出闭环 |
| 多轮扰动 | 单轮完成 | 用户追加约束 | 用户改变目标 / 工具失败 / 要求复用上下文 |

难度区间可以这样使用：

| 分数 | 难度 | 用途 |
| --- | --- | --- |
| 0–3 | 简单 | 冒烟测试，检查基础指令遵循 |
| 4–7 | 中等 | 主评估集主体，测组合能力 |
| 8–10 | 困难 | 拉开模型差距，测规划与判断 |
| 11–12 | 对抗 | 少量保留，用于边界测试，不宜占比过高 |

举三个例子：

| Query | 难度估计 | 原因 |
| --- | --- | --- |
| “帮我写一份电商售后话术模板。” | 1 分 | 几乎不依赖外部材料，也没有明显目标冲突 |
| “我上周买的羽绒服跑绒，超过 7 天但还在质保期；我想部分退款并保留商品，同时投诉商品页虚标面料。请给我协商话术、商家反驳预判、平台介入流程和证据清单。” | 7 分 | 有约束、有冲突、有判断，但还缺少可核验环境 |
| 同上，再加入订单 JSON、商品页快照、平台规则、同类判例，要求输出风险等级和优先级排序，且用户在第二轮补充“商家说质保期不包含人为损坏，怎么反驳”。 | 10–11 分 | 需要多材料交叉验证、多步规划和多轮状态追踪 |

评估集不是考试卷堆题，而是能力分辨率的设计。真正好的 query 应该能把模型拉开，而不是制造一片平分。如果所有模型都满分，说明题太简单；如果所有模型都崩，说明题可能过难或 grader 设置有问题。

## 6. 入库前 Checklist：先自查，再入库

合成 query 后，不要急着放进评估集。先过一遍 checklist。

| 基础有效性 | 通过标准 |
| --- | --- |
| 真实感 | 这句话像真实用户会说的，而不是考试题或教科书示例 |
| 任务性 | 它在完成具体任务，而不是泛泛归纳通用知识 |
| 交付物清晰 | 输出物有明确形式：报告、表格、话术、计划、清单、排序 |
| 可判分 | 至少有 2 个硬约束可以自动或半自动检查 |
| 可归因 | 模型失败后，能判断是信息缺失、约束遗漏、判断错误还是表达问题 |

| 结构完整性 | 检查要点 |
| --- | --- |
| prompt | 用户任务是否自然，读起来不像考试说明 |
| environment | 是否提供了完成任务所需的材料；材料和任务是否真正绑定 |
| hidden assumptions | 是否标注了哪些信息缺失、哪些只能假设 |
| hard_checks / soft_checks | 是否区分了必须通过的硬约束和可以评分的软质量 |
| grader | 是否有明确的 Pass/Fail 逻辑、评分 rubric 或人工复核规则 |

| 区分度信号 | 处理方式 |
| --- | --- |
| 强模型全部满分 | 增加约束冲突密度或加入多轮扰动 |
| 大多数模型完全无法完成 | 检查是否题目表述不清、环境材料不足或 grader 设置过严 |
| 人类专家也因信息不足无法判断 | 补充材料或降低对抗程度 |
| 明显像 LLM 合成出来的标准题 | 改写，加入真实业务摩擦感 |
| 评价只能得出“写得不错 / 一般” | 重写 hard_checks，补充可检查的硬约束 |

## 7. 一条完整 query 的样例

下面是一条可入库的非代码 eval query。每个字段都有它存在的理由：`environment` 是为了让模型无法仅凭通用知识作答；`hard_checks` 是为了让评估可归因；`failure_modes` 是为了让下一轮迭代有方向。

```yaml
id: ecommerce_refund_down_jacket_001
scenario: ecommerce_after_sales
difficulty_score: 9
difficulty_tags:
  - incomplete_information
  - policy_trap
  - multi_goal_conflict
  - risk_ranking

prompt: >
  我上周买的羽绒服，穿过一次发现跑绒，已经超过 7 天无理由，
  但还在质保期。我想申请部分退款并保留商品，同时投诉商家详情页虚标面料。
  请帮我整理跟客服协商的话术，预判商家可能的推脱理由并给出反驳；
  如果协商失败，再梳理平台介入流程和提前准备的凭证清单。
  所有建议请按高 / 中 / 低风险标注。

environment:
  - order_detail.json
  - product_page_snapshot.html
  - platform_refund_policy.md
  - dispute_precedent.md
  - evidence_checklist.xlsx

hard_checks:
  - 是否主动指出“部分退款并保留商品”不一定是平台默认支持的路径
  - 是否要求用户补充或核验关键证据：订单时间、商品页材质描述、跑绒照片/视频
  - 是否区分协商、投诉、平台介入三个阶段并分别给出行动项
  - 是否输出商家可能的推脱理由及对应反驳策略
  - 是否按高/中/低风险标注所有建议
  - 是否避免承诺“一定能部分退款成功”

soft_checks:
  - 话术是否有协商感，而不是激化对抗
  - 方案是否真正可执行，用户拿到后能直接使用
  - 风险提示是否与证据充分程度挂钩
  - 是否在用户诉求和平台规则之间做了合理取舍说明

failure_modes:
  - 直接套用通用退货话术模板，忽略本案特殊性
  - 忽略“超过 7 天”这一关键约束，默认用户仍在无理由退货期内
  - 承诺用户一定能争取到部分退款
  - 没有要求用户准备关键证据，如跑绒视频/照片
  - 只输出话术，不提供升级路径和证据清单
  - 风险标注缺失或标注不区分
```

这类 query 的价值不在于它长，而在于它能测出多个关键能力：规则理解、信息缺失处理、多目标冲突、风险判断、流程规划和可执行表达。

## 8. 评估集不是一次性题库，要有数据飞轮

非代码任务变化很快，模型也会随着迭代“学会”旧题型。一个健康的评估集需要持续进化。

可执行的数据飞轮可以分六步：

1. 线上捕获：记录用户点踩、重复追问、人工介入、工具失败、输出被大幅改写的 session。
2. 失败归因：把 badcase 标成几类，例如约束遗漏、事实错误、格式不合规、风险判断错误、没有主动追问、过度承诺、表达不符合场景。
3. 改写成 eval query：脱敏后保留真实任务压力，改写为标准结构：prompt、environment、expected behavior、grader。
4. 加入回归集：每次换模型、改 prompt、改工具链，都跑这批 query，确保旧问题不复发。
5. 定期退役：某类 query 所有主流模型连续多轮稳定通过后，从核心评估集降级到冒烟测试集。
6. 补充新难点：线上出现新失败模式时，立刻新增对应 query，比如“多轮压缩后遗漏关键卖点”“模型拒绝给合规维权建议”“工具超时后直接编造结果”。

维护时可以看这些信号：

| 信号 | 动作 |
| --- | --- |
| 某类 query 满分率超过 80% | 增加约束冲突或多轮扰动 |
| 某类 query 全部失败 | 检查题目表述、环境材料充分性和 grader 严格程度 |
| 线上 badcase 高频出现 | 加入核心回归集 |
| 题目长期无区分度 | 降级到冒烟测试 |
| 人工与 LLM judge 分歧持续偏大 | 重写 rubric 并增加锚定样例 |

这里的关键不是“越积越多”，而是让评估集保持分辨率。随着模型能力提升，评估集的难度边界也要同步上移。

## 写在最后

判断一条 query 是否值得入库，看五件事：

1. 它是否来自真实用户场景，而不是为了测试而设计的考试题。
2. 它是否有明确主考点，失败后能归因到具体能力维度。
3. 它是否包含可验证的硬约束，比如文字要求、数量限制、格式规范、禁止项。
4. 它是否有模型间区分度，不会所有模型都轻松通过或全部失败。
5. 它是否能暴露一种有业务意义的失败，而不是制造罕见刁钻题。

好的 query 不一定长，也不一定复杂。它的价值在于：一旦模型失败，我们能知道为什么失败；一旦模型进步，我们也能知道它在哪个能力上进步了。

每一条好的 query，都应该像一根探针，刺向模型最容易伪装成“已经会了”的地方。
